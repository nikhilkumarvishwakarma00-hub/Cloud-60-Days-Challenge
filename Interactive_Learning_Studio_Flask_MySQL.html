Foundations — REST, HTTP & Your First Flask Route

What REST actually means, and getting Flask to talk HTTP

Every web API — Twitter's, Razorpay's, the one you'll build for your Hogwarts Bank project — is just a program that speaks a very specific dialect: HTTP requests in, structured data out. REST is the etiquette for that conversation.

What "RESTful" really means

REST (Representational State Transfer) is a set of conventions, not a library. A RESTful API treats everything as a resource — a user, an account, a transaction — addressed by a URL, and manipulated using standard HTTP methods instead of custom verbs in the URL.

Analogy
Think of a REST API like a bank counter. You don't shout "PLEASE_GIVE_BALANCE" at the teller. You point at a form (the resource — your account) and choose an action from a fixed set: deposit, withdraw, check balance, close account. The teller (server) responds with a receipt (JSON) and a stamped status ("approved", "insufficient funds"). The action vocabulary is small and reused for every account — that's what makes it predictable.

Your first Flask route

Flask maps a URL + method pair to a Python function. That's the entire mental model to start.

from flask import Flask, jsonify

app = Flask(__name__)

@app.route("/api/v1/ping", methods=["GET"])
def ping():
    return jsonify({"status": "ok", "message": "pong"}), 200

if __name__ == "__main__":
    app.run(debug=True)

Notice three things: the @app.route decorator binds a URL to a function, jsonify() converts a Python dict into a proper JSON response with the right content-type header, and we return a tuple of (body, status_code).

Status codes — the API's tone of voice

2xx — Success

200 OK · 201 Created · 204 No Content

4xx — Client error

400 Bad Request · 401 Unauthorized · 404 Not Found · 409 Conflict

5xx — Server error

500 Internal Server Error · 503 Service Unavailable

3xx — Redirection

301 Moved Permanently · 304 Not Modified

Common misconception
"If my Flask code runs without crashing, I should return 200." No — the status code describes the outcome, not whether Python threw an exception. A validation failure that you catch gracefully should still return 400, not 200 with an error message buried in the body.

Key takeaways

  • REST maps operations onto standard HTTP methods against resource URLs — no custom action verbs in the path.
  • Flask binds (URL, method) pairs to Python functions via @app.route.
  • Status codes communicate outcome; pick the code that matches what actually happened, not just "did it crash."
  • jsonify() is the standard way to return JSON with correct headers in Flask.

Building a CRUD API in Flask

Routes, request parsing, validation, and JSON responses that don't fall over

A real API rarely has one route — it has a family of routes around one resource. Let's build the full CRUD set for an accounts resource, in memory first, so the HTTP layer is crystal clear before we bring in a database in Module 3.

The five routes of a resource

# in-memory store for now — Module 3 swaps this for MySQL
accounts = {}
next_id = 1

@app.route("/api/v1/accounts", methods=["GET"])
def list_accounts():
    return jsonify(list(accounts.values())), 200

@app.route("/api/v1/accounts/<int:acc_id>", methods=["GET"])
def get_account(acc_id):
    account = accounts.get(acc_id)
    if not account:
        return jsonify({"error": "Account not found"}), 404
    return jsonify(account), 200

@app.route("/api/v1/accounts", methods=["POST"])
def create_account():
    global next_id
    data = request.get_json(silent=True) or {}
    if not data.get("holder_name"):
        return jsonify({"error": "holder_name is required"}), 400
    account = {"id": next_id, "holder_name": data["holder_name"],
               "balance": data.get("balance", 0)}
    accounts[next_id] = account
    next_id += 1
    return jsonify(account), 201

@app.route("/api/v1/accounts/<int:acc_id>", methods=["PUT"])
def update_account(acc_id):
    if acc_id not in accounts:
        return jsonify({"error": "Account not found"}), 404
    data = request.get_json(silent=True) or {}
    accounts[acc_id].update(data)
    return jsonify(accounts[acc_id]), 200

@app.route("/api/v1/accounts/<int:acc_id>", methods=["DELETE"])
def delete_account(acc_id):
    if accounts.pop(acc_id, None) is None:
        return jsonify({"error": "Account not found"}), 404
    return "", 204
Analogy
Route parameters like <int:acc_id> are like a locker number on a locker room wall. The URL pattern /accounts/<int:acc_id> is the wall of lockers; the actual number the client sends picks one specific locker. Flask parses that segment straight into a Python int argument for you.

Validating input, the honest way

Never trust request.get_json() blindly. A client can send malformed JSON, missing fields, or wrong types. Always guard with silent=True plus explicit checks, and fail with 400 and a clear message — this is exactly the "honest framing over fabrication" habit that also makes for a better resume bullet: you can truthfully say you built input validation, not just happy-path routes.

Common misconception
"PUT and PATCH are basically the same, just use whichever." PUT implies a full replacement of the resource; PATCH implies a partial update. Using PUT for a partial update can silently wipe fields the client didn't send, if your handler naively replaces instead of merging.

Key takeaways

  • One resource → a consistent family of routes (list, get-one, create, update, delete).
  • Route converters like <int:acc_id> type-check and inject URL segments as function arguments.
  • Validate every incoming payload defensively; return 400 with a specific error message on failure.
  • PUT replaces, PATCH merges, DELETE returns 204 with an empty body on success.

Connecting Flask to MySQL

From an in-memory dict to a real, persistent, injection-safe data layer

Everything in Module 2 vanishes when the server restarts because accounts = {} lives in RAM. A real system — like your Hogwarts Bank Management System — needs a database that survives restarts and enforces structure. This is where MySQL comes in.

Setting up the connection

import mysql.connector
from mysql.connector import Error

def get_connection():
    return mysql.connector.connect(
        host="localhost",
        user="api_user",
        password="••••••••",   # load from env var, never hardcode
        database="bank_api"
    )
Common misconception
"It's fine to build SQL by joining strings, I'll just be careful." This is exactly how SQL injection happens. f"SELECT * FROM accounts WHERE id={acc_id}" lets a malicious acc_id like 1 OR 1=1 return every row in the table. Always use parameterized queries — the driver escapes values for you.

Naive string-building vs. parameterized queries

❌ Vulnerable

query = f"SELECT * FROM accounts WHERE id={acc_id}"
cursor.execute(query)

✅ Safe

query = "SELECT * FROM accounts WHERE id = %s"
cursor.execute(query, (acc_id,))

Wiring a real route to MySQL

@app.route("/api/v1/accounts/<int:acc_id>", methods=["GET"])
def get_account(acc_id):
    conn = get_connection()
    cursor = conn.cursor(dictionary=True)
    cursor.execute("SELECT * FROM accounts WHERE id = %s", (acc_id,))
    account = cursor.fetchone()
    cursor.close()
    conn.close()
    if not account:
        return jsonify({"error": "Account not found"}), 404
    return jsonify(account), 200

cursor(dictionary=True) is the detail that saves you a manual zip-with-column-names step — rows come back as dicts, ready for jsonify().

Key takeaways

  • Use mysql-connector-python (or SQLAlchemy) to bridge Flask routes to MySQL tables.
  • Always use parameterized queries (%s placeholders) — never f-strings — to prevent SQL injection.
  • cursor(dictionary=True) returns rows as dicts, pairing cleanly with jsonify().
  • Open a connection per request (or use a pool) and always close cursor + connection.

REST Design Principles & Production Readiness

Naming, versioning, pagination, auth — the difference between "it works" and "it's shippable"

A working CRUD API and a good API are different things. This module covers the conventions that make an API predictable to someone who has never seen your code — the kind of detail interviewers probe for when they ask "tell me about an API you built."

Resource naming conventions

GoodWhy
/api/v1/accountsPlural noun, no verb, versioned
/api/v1/accounts/104/transactionsNested resource shows ownership
/api/getAccount?id=104Verb in path, breaks REST convention
/api/v1/AccountInconsistent casing/pluralization

Versioning

Put a version in the URL (/api/v1/...) from day one. It costs nothing now and saves you from breaking every existing client when you inevitably need to change the response shape later.

Pagination for list endpoints

@app.route("/api/v1/accounts", methods=["GET"])
def list_accounts():
    page = request.args.get("page", 1, type=int)
    limit = request.args.get("limit", 20, type=int)
    offset = (page - 1) * limit
    cursor.execute("SELECT * FROM accounts LIMIT %s OFFSET %s", (limit, offset))
    rows = cursor.fetchall()
    return jsonify({"page": page, "limit": limit, "data": rows}), 200
Analogy
Pagination is a phone book, not a single scroll. Nobody hands over all 10,000 accounts in one response — you hand over "page 1 of 500," with a clear way to ask for the next page. It protects both your server's memory and the client's rendering time.

Basic authentication with an API key

from functools import wraps

def require_api_key(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        key = request.headers.get("X-API-Key")
        if key != os.environ.get("API_KEY"):
            return jsonify({"error": "Unauthorized"}), 401
        return f(*args, **kwargs)
    return decorated

@app.route("/api/v1/accounts", methods=["POST"])
@require_api_key
def create_account():
    ...

For anything beyond a class project, JWT-based auth (issuing a signed token on login) is the industry default — the pattern above shows the gate mechanism a JWT check would slot into.

Common misconception
"My API works when I test it in Postman, so it's production-ready." Postman tests the happy path you thought of. Production readiness means: consistent error shapes, no secrets in code, pagination on lists, rate-limit awareness, and versioning — none of which show up until someone else, or scale, stresses the system.

Design principle comparison

PrincipleWeak versionStrong version
ErrorsPlain string message{"error": "...", "code": "ACCOUNT_NOT_FOUND"}
FilteringReturn everything, filter client-side?status=active&min_balance=1000
ConsistencyMixed casing, ad-hoc shapessnake_case fields, uniform envelope

Key takeaways

  • Use plural nouns, no verbs, and consistent nesting for resource URLs.
  • Version your API from the first commit: /api/v1/...
  • Paginate every list endpoint with page/limit query params.
  • Gate write operations behind an auth check; never hardcode secrets.
  • Consistent, structured error responses are as much a design decision as your success responses.

Summary notes

REST turns state changes into predictable HTTP verbs against resource URLs. Flask gives you the routing and JSON layer; MySQL gives you persistence, provided every query is parameterized. Production-credible APIs add versioning, pagination, structured errors, and an auth gate on top of working CRUD — that layer is usually what separates a class project from a resume-worthy one.
